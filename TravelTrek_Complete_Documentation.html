Introduction & Project Overview

TravelTrek is an AI-powered travel planning mobile application developed as a graduation project. It enables users to plan personalized travel itineraries using natural language prompts, track budgets, view day-by-day activities, and manage their travel plans seamlessly from a mobile device.

The application integrates a custom RESTful backend API hosted at traveltrek.runasp.net with Google Firebase Authentication, providing a hybrid authentication strategy that supports both email/password and Google Sign-In.

🎯 Core Value Proposition: Users simply describe their dream trip in one sentence, and TravelTrek's AI backend generates a complete, multi-day itinerary with activities, meals, weather info, packing tips, and budget estimates — all personalized to the user's preferences and local currency.

1.2 Application Scope

TravelTrek targets mobile platforms (Android & iOS) as primary targets, with Flutter also making the code portable to web, Windows, macOS, and Linux. The app is designed for individual travelers and families planning short to medium-duration trips (1–14 days).

Architecture Design

3.1 Clean Architecture Pattern

TravelTrek follows Clean Architecture (Robert C. Martin). Every feature module is divided into three independent layers:

🖥️ Presentation Views · Widgets · BLoC Cubits · States
↓ depends on
📐 Domain Entities · Repository Interfaces · Business Logic
↓ implements
🗄️ Data Models (fromJson/toJson) · Repository Implementations · API Calls
↑ shared by all
⚙️ Core Services · Utilities · Shared Widgets · Error Types · DI Setup
LayerResponsibilityKey Rule
PresentationUI rendering, user interaction, state via CubitNo data source knowledge
DomainBusiness logic, entities, abstract repo contractsPure Dart — zero framework imports
DataAPI calls, JSON parsing, concrete repository implementationsImplements domain contracts
CoreShared utilities, DI, colors, styles, routingCross-cutting concerns only

3.2 BLoC / Cubit State Management

TravelTrek uses Flutter BLoC's Cubit pattern throughout. A Cubit is a simplified BLoC without explicit Events — state changes are triggered by directly calling methods. Every repository call returns Either<Failure, T> from the dartz package, ensuring explicit error handling.

// Standard Cubit state pattern used across all features
abstract class SignInState {}
class SignInInitial extends SignInState {}  // Default
class SignInLoading extends SignInState {}  // Show spinner
class SignInSuccess extends SignInState { final UserEntity userEntity; }
class SignInFailure extends SignInState { final String errorMessage; }

3.3 Dependency Injection with GetIt

All singleton services and repositories are registered at app startup via the setup() function in core/services/service_locator.dart.

final getIt = GetIt.instance;

void setup() {
  getIt.registerSingleton<FirebaseAuthService>(FirebaseAuthService());
  getIt.registerSingleton<Api>(Api(dio: Dio(BaseOptions(
    connectTimeout: const Duration(seconds: 60),
    receiveTimeout: const Duration(seconds: 60),
  ))));
  getIt.registerSingleton<AuthRepo>(
    AuthRepoImpleApi(api: getIt<Api>(), firebaseAuthService: getIt<FirebaseAuthService>()),
  );
  getIt.registerSingleton<ExpenseRepo>(ExpenseRepoImple(apiService: getIt<Api>()));
  getIt.registerSingleton<HomeRepo>(HomeRepoImple(apiService: getIt<Api>()));
  getIt.registerSingleton<PlanRepo>(PlanRepoImple(apiService: getIt<Api>()));
  getIt.registerSingleton<PlansRepo>(PlansRepoImple(apiService: getIt<Api>()));

The authentication feature handles all user identity operations using a hybrid approach: email/password goes through the custom REST API, while Google Sign-In flows through Firebase, with the resulting session stored in SharedPreferences.

9.2 Sign Up Screen

Collects full name, email, and password. Validates form then calls SignUpCubit.signUp() which posts to POST /api/auth/register. Includes Terms & Conditions checkbox and navigation link to Sign In.

9.3 Google Sign-In

Implemented using google_sign_in + firebase_auth. The flow: initialize GoogleSignIn → trigger OAuth consent → get ID token → create Firebase credential → sign in to Firebase → convert to UserModel → persist to SharedPreferences.

9.4 Change Password

The ChangePasswordCubit calls FirebaseAuthService.changePassword(), which re-authenticates the user with their current credentials before updating to the new password. This prevents unauthorized password changes on shared devices.

Data Layer Overview

The data layer is responsible for all network communication and JSON serialization. It implements abstract repository interfaces from the domain layer, keeping all layers fully decoupled.

Key Pattern: Every repository method returns Future<Either<Failures, T>>. The Either monad (from dartz) forces callers to handle both success (Right) and failure (Left) paths explicitly — eliminating silent failures.

class UserEntity {
  final String name;
  final String email;
  final String uId;
  final String token;   // JWT access token from backend
}

// Data layer: UserModel extends UserEntity
class UserModel extends UserEntity {
  factory UserModel.fromJson(json) => UserModel(
    name:  json['value']['user']['fullName'],
    email: json['value']['user']['email'],
    uId:   json['value']['user']['id'],
    token: json['value']['accessToken'],
  );
  factory UserModel.fromFireBaseUser({required User user}) => UserModel(
    name: user.displayName ?? 'Alex',
    email: user.email ?? '',
    uId: user.uid,
    token: user.phoneNumber ?? '',
  );
}

Firebase Authentication Service

The FirebaseAuthService class isolates all Firebase SDK calls from the rest of the app. It provides a clean interface returning Dart native types and throws CustomException for all error cases.

Design Decision: Isolating Firebase in this service means AuthRepo has zero Firebase imports. The entire repository can be tested with mock services without a real Firebase project.

Email/Password Authentication

Both createUserWithEmailAndPassword() and signInWithEmailAndPassword() wrap Firebase SDK calls, catch FirebaseAuthException, map specific error codes to user-friendly messages, and throw CustomException for all other cases.

Google Sign-In Flow

Future<User> signInWithGoogle() async {
  final googleSignIn = GoogleSignIn.instance;
  await googleSignIn.initialize();                    // 1. Initialize
  final googleUser = await googleSignIn.authenticate(); // 2. OAuth consent
  final credential = GoogleAuthProvider.credential(     // 3. Firebase credential
    idToken: googleUser.authentication.idToken,
  );
  return (await FirebaseAuth.instance.signInWithCredential(credential)).user!;
}

Change Password with Re-authentication

Requires re-authentication before password update to prevent unauthorized changes on shared devices:

final credential = EmailAuthProvider.credential(email: email, password: oldPassword);
await user.reauthenticateWithCredential(credential);   // Verify old password
await user.updatePassword(newPassword);               // Update to new password

Currency & Location Detection Service

The CurrencyLocationService automatically detects the user's local currency by combining GPS with reverse geocoding, with graceful fallbacks:

Step 1: Check if location services are enabled
Step 2: Request location permission (if denied → fallback to locale)
Step 3: Get GPS coordinates (10-second timeout)
Step 4: Reverse geocode coordinates → ISO country code (10-second timeout)
Step 5: Map country code → currency symbol
Fallback 1: Device locale (PlatformDispatcher.locale.countryCode)
Fallback 2: Default = SAR (Saudi Riyal)
