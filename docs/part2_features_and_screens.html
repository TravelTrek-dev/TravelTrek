Splash Screen

✈️

SplashView

Features/splash/presentation/views/splash_view.dart

The splash screen is the entry point of the application. It is configured as the initialRoute in the MaterialApp and performs two important checks before routing the user:

App Launches → SplashView
  ├─ Read: kIsOnBoardingSeen from SharedPreferences
  │   ├─ false → Navigate to OnBoardingView
  │   └─ true ↓
  └─ Read: kUserEntity from SharedPreferences
      ├─ null → Navigate to SignInView
      └─ exists → Navigate to MainView

The splash screen also displays the TravelTrek logo and brand identity during the brief initialization period. It uses a Future.delayed or initState logic to check conditions after the first frame renders.

Authentication Feature

The authentication feature handles all user identity operations. It uses a hybrid approach: email/password authentication goes through the custom REST API, while Google Sign-In flows through Firebase Authentication, with the resulting user session stored locally via SharedPreferences.

3.1 Sign In Screen

🔐

SignInView + SignInViewBody

Features/auth/presentation/views/sign_in_view.dart

The sign-in screen provides email/password login and Google Sign-In options. It uses a BlocConsumer<SignInCubit, SignInState> to react to state changes — showing a loading indicator on SignInLoading, navigating to MainView on success, and displaying a SnackBar on failure.

Sign In Flow

// State machine in SignInCubitState
abstract class SignInState {}
class SignInInitial  extends SignInState {}  // Default
class SignInLoading  extends SignInState {}  // Show spinner
class SignInSuccess  extends SignInState { final UserEntity userEntity; }
class SignInFailure  extends SignInState { final String errorMessage; }
Remember Me: The sign-in screen saves the user token and entity to SharedPreferences (kUserToken, kUserEntity) upon successful login, enabling automatic authentication on subsequent app launches.

3.3 Google Sign-In

🔵

Google OAuth via Firebase

core/services/firebase_auth_service.dart

Google Sign-In is implemented using the google_sign_in and firebase_auth packages. The flow is triggered by SignInCubit.signInWithGoogle() and delegates to FirebaseAuthService.signInWithGoogle().

Future<User> signInWithGoogle() async {
  final googleSignIn = GoogleSignIn.instance;
  await googleSignIn.initialize();
  final googleUser = await googleSignIn.authenticate();

  final credential = GoogleAuthProvider.credential(
    idToken: googleUser.authentication.idToken,
  );
  return (
    await FirebaseAuth.instance.signInWithCredential(credential)
  ).user!;
}

On success, a UserModel.fromFireBaseUser() converts the Firebase User object into a UserEntity, which is then persisted locally.

3.4 Change Password

Users can change their password from the Profile screen. The ChangePasswordCubit calls FirebaseAuthService.changePassword(), which re-authenticates the user with their old credentials before updating to the new password. This prevents unauthorized password changes on shared devices.

Home Screen

🏠

HomeView & HomeViewBody

Features/home/presentation/views/widgets/

The home screen is the core user-facing feature of TravelTrek. It is hosted inside the MainView as the first tab of the bottom navigation bar. It gives the user the ability to describe their dream trip in natural language and immediately generate an AI-powered plan.

Layout Structure

HomeView
└─ BlocProvider(GeneratePlanCubitCubit)
   └─ HomeViewBody (StatefulWidget)
      ├─ BlocListener (navigate on success/show error on failure)
      └─ SingleChildScrollView
         ├─ Text: "Describe Your Dream Trip"
         ├─ TripPlannerInputField
         ├─ "RECENT" label + RecentPlanListViewBuilderBlocBuilder
         └─ CustomButton: "Generate My Plan"

4.1 AI Prompt Input

🤖

TripPlannerInputField

The TripPlannerInputField is a multi-line text area where users type their travel request. Below the text field are quick-fill chips (built by BuildChip widget) for common trip types like "Family vacation", "Beach holiday", "Mountain trek", etc.

When the user taps "Generate My Plan", the cubit is invoked:

BlocProvider.of<GeneratePlanCubitCubit>(context).generatePlan(
  userToken: Prefs.getString(kUserToken),
  promot: _promptController.text.trim(),
);

The cubit calls HomeRepo.generatePlan(), which sends a GET request to:

GET http://traveltrek.runasp.net/api/trip-plan/generate?prompt={text}&currency={userCurrency}
Auth: Bearer token in Authorization header

4.2 Recent Plans Section

The Recent Plans horizontal list fetches the user's prompt history from the backend. Each RecentPlanItem card shows a trip destination thumbnail and destination name. Tapping a recent prompt re-fills the input field, letting the user regenerate with the same prompt.

WidgetPurpose
RecentPlanListViewBuilderBlocBuilderBlocBuilder wrapper; shows loading/empty/list states
RecentPlanListViewBuilderHorizontal ListView of plan prompt cards

ItineraryView

Each DayCard represents one day of the trip. It uses a expandable/collapsible design showing the day number, meals summary, and a list of activities.

class Activity extends Equatable {
  final String? name;
  final String? description;
  final String? googleMapsLink;   // Deep link to Google Maps
  final dynamic website;            // Official website URL
  final String? city;
  final String? approximateCost;  // e.g. "$15 – $30"
  final String? type;              // e.g. "sightseeing", "food"
}

5.3 Travel Tips Section

Below the daily itinerary, the view shows:

  • LocalCurrencyCard: shows the user's detected currency with an exchange rate card
  • General Advice: a paragraph of AI-generated travel advice specific to the destination
  • Packing Tips: bulleted list of items to pack

5.4 Save Plan

The "Save Plan" button at the bottom calls PlanDetailsCubit.savePlan(), which sends the plan data to the backend POST /api/trip-plan/save-created?token={token}. A SnackBar confirms success or shows the error message.

Budget Tracker Feature

The budget tracking feature allows users to monitor their real spending against the AI-generated budget estimate. It is divided into two primary screens: the dashboard and the expense entry form.

7.1 Budget Dashboard

💰

BudgetView & BudgetViewBody

Features/Budget/presentation/views/

The budget dashboard presents a real-time financial overview of the trip spending. It receives the plan's budget from the PlanModel and fetches expense records from the backend to compute spent/remaining amounts.

// Budget calculation logic in BudgetViewBody
double totalBudget = planModel.value!.budget!.toDouble();
double totalSpent = expenses.fold(0.0, (sum, e) => sum + e.price);
double remaining = totalBudget - totalSpent;
double utilizedPercent = (totalSpent / totalBudget).clamp(0.0, 1.0);

// Logout flow
Future<void> _logout(BuildContext context) async {
  await Prefs.removeUserEntity();  // Clear stored user
  Navigator.pushNamedAndRemoveUntil(
    context,
    SignInView.routeName,
    (route) => false,            // Clear all routes
  );
}
