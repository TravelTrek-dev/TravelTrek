Data Layer Overview

The data layer is the lowest application layer responsible for all network communication and JSON serialization/deserialization. It implements the abstract repository interfaces defined in the domain layer, keeping the domain and presentation layers completely decoupled from implementation details.

Key Pattern: Every repository method returns Future<Either<Failures, T>>. The Either monad (from the dartz package) forces callers to handle both success (Right) and failure (Left) paths explicitly. This eliminates silent failures.

Data Layer Responsibilities

ResponsibilityHow
HTTP CommunicationVia the Api class (Dio wrapper) with Bearer token auth
JSON DeserializationModel.fromJson(Map) factory constructors
JSON Serializationmodel.toJson() methods returning Map<String, dynamic>
Error TranslationConverting DioException / CustomException into ServerFailure
Domain MappingData models extend domain entities; data classes are passed upward as entities

Domain Entities

2.1 UserEntity

The UserEntity is the core domain object representing an authenticated user. It is defined in the domain layer and has no dependencies on external packages.

// lib/Features/auth/domain/entity/user_entity.dart
class UserEntity {
  final String name;
  final String email;
  final String uId;
  final String token;  // JWT access token from backend

  const UserEntity({
    required this.name,
    required this.email,
    required this.uId,
    required this.token,
  });

  // JSON serialization for SharedPreferences storage
  Map<String, dynamic> toJson() => {
    'name': name, 'email': email,
    'uId': uId, 'token': token,
  };

  factory UserEntity.fromJson(Map<String, dynamic> json) => UserEntity(
    name: json['name'], email: json['email'],
    uId: json['uId'], token: json['token'],
  );
}

The UserModel extends UserEntity in the data layer and adds two factory constructors: one for Firebase User objects (Google Sign-In) and one for backend API JSON responses.

Factory ConstructorSourceField Mapping
UserModel.fromJson(json)Backend API responsejson['value']['user']['fullName'] → name
json['value']['accessToken'] → token
UserModel.fromFireBaseUser(user)Firebase Auth Useruser.displayName → name
user.uid → uId

2.3 ExpenseEntity & ExpenseModel

The expense entity represents a single logged expense item within a trip plan.

class ExpenseEntity {
  final String description;  // e.g. "Dinner at Eiffel Tower restaurant"
  final double price;        // Amount spent
  final String category;    // e.g. "food", "transport"
  final double? spent;       // Total spent in category (from API)
  final double? remaining;   // Remaining budget in category (from API)
}

// Data layer model with JSON support
class ExpenseModel extends ExpenseEntity {
  Map<String, dynamic> toJson() => {
    'description': description, 'price': price, 'category': category,
  };

  factory ExpenseModel.fromJson(Map<String, dynamic> json) => ExpenseModel(
    description: json['description'],
    price: json['price'].toDouble(),
    category: json['name'],
    spent: json['spent'] ?? 0.0,
    remaining: json['remaining'] ?? 0.0,
  );
}

Repository Pattern

Each feature follows the Repository Pattern: an abstract interface in the domain layer defines the contract, and a concrete implementation in the data layer fulfills it. This separation enables easy mocking for tests and future backend replacements.

Error Handling Architecture

TravelTrek uses a two-tier error handling system: CustomException for low-level errors that propagate within the data layer, and Failures (sealed class) for domain-safe errors that flow up to the presentation layer.

// lib/core/errors/custom_exception.dart
class CustomException implements Exception {
  final String message;
  const CustomException({required this.message});
}

// lib/core/errors/failures.dart
abstract class Failures {
  final String errorMessage;
  const Failures({required this.errorMessage});
}

class ServerFailure extends Failures {
  const ServerFailure({required super.errorMessage});
}

API HTTP Client

The Api class (lib/core/helper_function/api.dart) is a thin wrapper around the Dio HTTP client. It provides three methods — get, post, and put — each of which automatically handles:

  • Setting the Authorization: Bearer {token} header when a token is provided
  • Catching DioException and rethrowing as CustomException with a user-friendly message
  • Extracting error messages from structured API response bodies (response.data['message'])
// lib/core/helper_function/api.dart
class Api {
  final Dio dio;

  Future<dynamic> get({required String url, String? token}) async {
    final headers = token != null
        ? {'Authorization': 'Bearer $token'}
        : <String, String>{};
    try {
      final response = await dio.get(url, options: Options(headers: headers));
      return response.data;
    } on DioException catch (e) {
      final message = e.response?.data?['message'];
      throw CustomException(
        message: message ?? e.message ?? 'Connection error'
      );
    }
  }
  // post() and put() follow same pattern
}
