Firebase Authentication Service

The FirebaseAuthService class (lib/core/services/firebase_auth_service.dart) is a dedicated service layer that wraps the Firebase Authentication SDK. It abstracts all Firebase-specific logic away from the repository and domain layers, providing a clean interface that returns Dart's native types (User or throws CustomException).

Design Decision: By isolating Firebase calls in this service, the AuthRepo interface has no Firebase imports. This means the entire auth repository can be tested with mock services without needing a real Firebase project.

1.1 Email/Password Authentication

📧 createUserWithEmailAndPassword(name, email, password)

Creates a new Firebase user account. If successful, it internally calls the backend register API (via AuthRepoImpleApi) to create the server-side user record. The Firebase account creation is currently used for credential validation — the primary user creation goes through the backend REST API.

Firebase Error CodeAction
weak-passwordThrows CustomException("The password provided is too weak.")
email-already-in-useThrows CustomException("The account already exists for that email.")
Any other errorThrows CustomException("Error. pls try again later!")

🔑 signInWithEmailAndPassword(email, password)

Authenticates an existing user with Firebase. Note that in the final implementation, sign-in primarily goes through the backend REST API (/api/auth/login) which returns the JWT token. Firebase sign-in is used as a secondary credential verification layer.

Firebase Error CodeUser-Facing Message
user-not-found"password or email not valid"
wrong-password"password or email not valid"
invalid-credential"password or email not valid"

1.2 Google Sign-In Flow

🔵 signInWithGoogle()

Implements the full Google OAuth flow using the GoogleSignIn package combined with Firebase Authentication. The flow is entirely handled within this service:

Future<User> signInWithGoogle() async {
  // 1. Initialize Google Sign-In
  final googleSignIn = GoogleSignIn.instance;
  await googleSignIn.initialize();

  // 2. Trigger OAuth consent screen
  final googleUser = await googleSignIn.authenticate();

  // 3. Get ID token from Google
  final googleAuth = googleUser.authentication;

  // 4. Create Firebase credential
  final credential = GoogleAuthProvider.credential(
    idToken: googleAuth.idToken,
  );

  // 5. Sign in to Firebase with Google credential
  return (await FirebaseAuth.instance.signInWithCredential(credential)).user!;
}

On success, the returned Firebase User object is converted to a UserModel using the UserModel.fromFireBaseUser() factory, then persisted to SharedPreferences.

1.3 Change Password with Re-authentication

🔒 changePassword(email, oldPassword, newPassword)

Changing a password is a sensitive operation that Firebase requires recent authentication for. The service implements re-authentication before the password update to ensure security:

Future<void> changePassword({email, oldPassword, newPassword}) async {
  final user = FirebaseAuth.instance.currentUser;
  if (user == null) throw CustomException(message: 'Please sign in again.');

  // Re-authenticate with current credentials first
  final credential = EmailAuthProvider.credential(
    email: email, password: oldPassword,
  );
  await user.reauthenticateWithCredential(credential);

  // Now safely update the password
  await user.updatePassword(newPassword);
}
Security Note: Re-authentication ensures that even if a device is left unlocked, an attacker cannot change the password without knowing the current password. This follows Firebase's security best practices for sensitive account operations.

Local Storage — Prefs Service

The Prefs class (lib/core/services/prefs.dart) is a static wrapper around Flutter's SharedPreferences package, providing a simple, type-safe API for persisting app data locally on the device.

It is initialized asynchronously at app startup in main() and stored as a late static instance (static late SharedPreferences _instance), making all subsequent calls synchronous.

Currency & Location Detection Service

The CurrencyLocationService (lib/core/services/currency_location_service.dart) automatically detects the user's local currency by combining GPS coordinates with reverse geocoding. This currency is then passed to the AI plan generation API so the generated budget is always in the user's local currency.

3.1 Currency Detection Flow

Future<String> getUserCurrency() async {
  // Step 1: Check if location services are enabled
  bool serviceEnabled = await Geolocator.isLocationServiceEnabled();
  
  // Step 2: Request location permission if not granted
  var permission = await Geolocator.checkPermission();
  if (permission == LocationPermission.denied) {
    permission = await Geolocator.requestPermission();
  }

  // Step 3: Fallback to locale if permission denied
  if (permission == LocationPermission.denied ||
      permission == LocationPermission.deniedForever) {
    return _getCurrencyFromLocale();
  }

  // Step 4: Get GPS coordinates (10-second timeout)
  final position = await Geolocator.getCurrentPosition(
    locationSettings: const LocationSettings(accuracy: LocationAccuracy.low),
  ).timeout(const Duration(seconds: 10));

  // Step 5: Reverse geocode coordinates → country code
  final placemarks = await placemarkFromCoordinates(
    position.latitude, position.longitude,
  ).timeout(const Duration(seconds: 10));

  // Step 6: Map country code → currency symbol
  final countryCode = placemarks.first.isoCountryCode;
  return _getCurrencyFromCountryCode(countryCode);
}

Fallback Strategy

The service implements a graceful degradation strategy with three fallback levels:

  1. GPS + Geocoding (primary) — Most accurate, uses actual device location
  2. Device Locale (fallback) — Uses ui.PlatformDispatcher.instance.locale.countryCode
  3. Default: SAR — Saudi Riyal, the app's default currency when all detection fails

BLoC Observer (Debug Logging)

The SimpleBlocObserver (lib/core/services/simple_bloc_observer.dart) extends Flutter BLoC's BlocObserver to provide centralized logging of all state transitions, errors, and events during development.

// lib/core/services/simple_bloc_observer.dart
class SimpleBlocObserver extends BlocObserver {
  @override
  void onCreate(BlocBase bloc) {
    super.onCreate(bloc);
    log('onCreate -- ${bloc.runtimeType}');
  }

  @override
  void onChange(BlocBase bloc, Change change) {
    super.onChange(bloc, change);
    log('onChange -- ${bloc.runtimeType}, $change');
  }

  @override
  void onError(BlocBase bloc, Object error, StackTrace stackTrace) {
    log('onError -- ${bloc.runtimeType}, $error');
    super.onError(bloc, error, stackTrace);
  }
}

It is registered globally in main() before the app starts:

Bloc.observer = SimpleBlocObserver();
Development Benefit: The observer helps trace exactly which cubits are transitioning between states and when errors occur, making debugging state management issues significantly faster during development.

Dependency Injection Setup

TravelTrek uses GetIt as a Service Locator for dependency injection. All services and repositories are registered as singletons — meaning one instance is shared across the entire app lifecycle.

App Initialization Sequence

The main() function in lib/main.dart performs a critical sequence of async initialization steps before the Flutter UI is rendered:

void main() async {
  // Step 1: Ensure Flutter engine is initialized
  WidgetsFlutterBinding.ensureInitialized();

  // Step 2: Register global BLoC observer for debugging
  Bloc.observer = SimpleBlocObserver();

  // Step 3: Initialize Firebase with platform-specific config
  await Firebase.initializeApp(
    options: DefaultFirebaseOptions.currentPlatform,
  );

  // Step 4: Register all singleton services and repositories
  setup();

  // Step 5: Initialize SharedPreferences
  await Prefs.init();

  // Step 6: Read and log saved currency for debugging
  final localCurrency = Prefs.getString(kSavedCurrency);
  log('Saved Local Currency: $localCurrency');

  // Step 7: Launch the app
  runApp(const TravelTrek());
}

Firebase Configuration

TravelTrek uses Firebase for authentication services. The configuration is managed by the auto-generated firebase_options.dart file, created using the FlutterFire CLI (flutterfire configure).

⚠️ Security Warning: The firebase_options.dart file contains API keys and app IDs. While Firebase API keys for client apps are not secret by themselves (they are restricted by Firebase Security Rules and SHA certificate fingerprints), this file should still be excluded from public repositories via .gitignore.

Build & Deployment Guide

8.1 Android Build

Prerequisites

  • Android Studio with Android SDK API level 21+ (Android 5.0 Lollipop or higher)
  • google-services.json placed at android/app/google-services.json
  • SHA-1 certificate fingerprint added to Firebase Console for Google Sign-In
  • Internet permission declared in AndroidManifest.xml
  • Location permissions for CurrencyLocationService: ACCESS_FINE_LOCATION and ACCESS_COARSE_LOCATION

Build Commands

# Development build (debug APK)
flutter build apk --debug

# Release APK (unsigned)
flutter build apk --release

# Release APK (split per ABI — smaller file sizes)
flutter build apk --split-per-abi --release

# App Bundle (recommended for Play Store)
flutter build appbundle --release

Android Manifest Permissions

<!-- Required for API calls -->
<uses-permission android:name="android.permission.INTERNET" />

<!-- Required for CurrencyLocationService -->
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_COARSE_LOCATION" />

8.2 iOS Build

Prerequisites

  • macOS with Xcode 14+ and valid Apple Developer account
  • GoogleService-Info.plist placed in ios/Runner/
  • Bundle ID registered in Firebase Console
  • Location usage descriptions in Info.plist

Build Commands

# iOS debug build
flutter build ios --debug

# iOS release build (requires Apple Developer account)

# IPA for App Store distribution

User Data Storage UserEntity JSON stored in SharedPreferences ⚠️ Move to secure storage for sensitive fields
