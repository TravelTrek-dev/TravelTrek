Introduction

1.1 Project Overview

TravelTrek is an AI-powered travel planning mobile application developed as a graduation project. It enables users to plan personalized travel itineraries using natural language prompts, track budgets, view day-by-day activities, and manage their travel plans seamlessly from a mobile device.

The application integrates a custom RESTful backend API hosted at traveltrek.runasp.net with Google Firebase Authentication, providing a hybrid authentication strategy that supports both email/password and Google Sign-In.

🎯 Core Value Proposition: Users simply describe their dream trip in one sentence, and TravelTrek's AI backend generates a complete, multi-day itinerary with activities, meals, weather info, packing tips, and budget estimates — all personalized to the user's preferences and local currency.

1.3 Application Scope

TravelTrek targets mobile platforms (Android & iOS) as primary targets, with the Flutter framework also making the code portable to web, Windows, macOS, and Linux. The app is designed for individual travelers and families planning short to medium-duration trips (1–14 days).

Architecture

3.1 Clean Architecture

TravelTrek follows the Clean Architecture pattern popularized by Robert C. Martin ("Uncle Bob"), adapted for Flutter. Every feature module is divided into three independent layers:

🖥️ Presentation Views (Screens)  ·  Widgets  ·  BLoC Cubits  ·  States
↓ depends on
📐 Domain Entities  ·  Repository Interfaces  ·  Use Cases
↓ depends on (for impl)
🗄️ Data Models (from/toJson)  ·  Repository Implementations  ·  Data Sources (API)
↑ shared by all
⚙️ Core Services  ·  Utils  ·  Widgets  ·  Error Handling  ·  DI Setup

Layer Responsibilities

LayerResponsibilityKey Principle
PresentationUI rendering, user interaction, state management via CubitKnows nothing about data sources
DomainBusiness logic, entity definitions, abstract repository contractsPure Dart; zero framework imports
DataAPI calls, JSON serialization, concrete repository implementationsImplements domain contracts
CoreShared utilities, DI, colors, styles, routing helpersCross-cutting concerns only

3.2 BLoC / Cubit State Management

TravelTrek uses Flutter BLoC's Cubit pattern throughout. A Cubit is a simplified BLoC without explicit Events — state changes are triggered directly by calling methods on the Cubit. This keeps boilerplate low while retaining reactive, testable state.

Every feature cubit follows this pattern:

// Example: Sign-In Cubit structure
class SignInCubit extends Cubit<SignInState> {
  final AuthRepo authRepo;

  SignInCubit({required this.authRepo}) : super(SignInInitial());

  Future<void> signIn({required String email, required String password}) async {
    emit(SignInLoading());
    final result = await authRepo.signInWithEmailAndPassword(
      email: email, password: password,
    );
    result.fold(
      (failure) => emit(SignInFailure(errorMessage: failure.errorMessage)),
      (user) => emit(SignInSuccess(userEntity: user)),
    );
  }
}

// States: Initial → Loading → Success | Failure

The Either<Failure, T> type from dartz package ensures every repository call returns either an error (left) or success data (right), enforcing explicit error handling everywhere.

3.3 Dependency Injection

TravelTrek uses GetIt as a service locator. All singleton services and repository implementations are registered at app startup in the setup() function located in lib/core/services/service_locator.dart.

// lib/core/services/service_locator.dart
final GetIt getIt = GetIt.instance;

void setup() {
  // Core services
  getIt.registerSingleton<FirebaseAuthService>(FirebaseAuthService());
  getIt.registerSingleton<Api>(Api(dio: Dio(BaseOptions(
    connectTimeout: const Duration(seconds: 60),
    receiveTimeout: const Duration(seconds: 60),
  ))));

  // Feature repositories
  getIt.registerSingleton<AuthRepo>(
    AuthRepoImpleApi(api: getIt<Api>(), firebaseAuthService: getIt<FirebaseAuthService>()),
  );
  getIt.registerSingleton<ExpenseRepo>(ExpenseRepoImple(apiService: getIt<Api>()));
  getIt.registerSingleton<HomeRepo>(HomeRepoImple(apiService: getIt<Api>()));
  getIt.registerSingleton<PlanRepo>(PlanRepoImple(apiService: getIt<Api>()));
  getIt.registerSingleton<PlansRepo>(PlansRepoImple(apiService: getIt<Api>()));
}
Design Benefit: Using GetIt as a service locator means cubits, repositories, and services are fully decoupled. Swapping a FirebaseAuthService implementation for a mock during testing requires changing only one registration line in service_locator.dart.

Navigation System

TravelTrek uses Flutter's built-in named route navigation with a custom onGenerateRoute function. Route names are defined as static constants on each view class, preventing typo errors.

// lib/core/helper_function/on_generate_route.dart
Route<dynamic> onGenerateRoute(RouteSettings setting) {
  switch (setting.name) {
    case SplashView.routeName:
      return MaterialPageRoute(builder: (_) => const SplashView());
    case OnBoardingView.routeName:
      return MaterialPageRoute(builder: (_) => const OnBoardingView());
    case SignInView.routeName:
      return MaterialPageRoute(builder: (_) => const SignInView());
    case SignUpView.routeName:
      return MaterialPageRoute(builder: (_) => const SignUpView());
    case MainView.routeName:
      return MaterialPageRoute(builder: (_) => const MainView());
    case PlanViewDetails.routeName:
      final planModel = setting.arguments as PlanModel?;
      return MaterialPageRoute(builder: (_) => PlanViewDetails(planModel: planModel!));
    case ProfileView.routeName:
      return MaterialPageRoute(builder: (_) => const ProfileView());
    default:
      return MaterialPageRoute(builder: (_) => const Scaffold());
  }
}
